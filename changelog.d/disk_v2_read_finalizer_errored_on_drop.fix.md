Fixed a `disk_v2` buffer bug where an acknowledged event could be permanently lost after a crash. A record read from the buffer carries a finalizer downstream; if that finalizer was dropped without an acknowledgement — for example, when the sink task is torn down with the delivery in flight around a crash — it finalized with the default "dropped" status, which the batch treated as a no-op, leaving the batch at its optimistic `Delivered` default. The buffer then advanced its read position past a record that was never delivered, and on reopen that record (durable on disk) was skipped and lost.

The read path now attaches a pessimistic finalizer, so a dropped, unacknowledged delivery is reported as a failure rather than silently counted as delivered (mirroring the write path's existing errored-on-drop guard). On such a failure the buffer stops advancing its acknowledgement frontier, so the record — and any read after it — is re-read and re-delivered when the buffer is reopened, preserving at-least-once delivery. A permanent downstream rejection is still advanced past (and recorded), since it can never be delivered.
